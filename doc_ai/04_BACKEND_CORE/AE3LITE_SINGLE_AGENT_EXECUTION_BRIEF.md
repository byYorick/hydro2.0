# AE3-Lite v1: Single-Agent Execution Brief

**Версия:** 1.0
**Дата:** 2026-03-06
**Статус:** EXECUTION BRIEF

Compatible-With: Protocol 2.0, Backend >=3.0, Python >=3.0, Database >=3.0, Frontend >=3.0.

---

## 0. Назначение

Этот документ читает один ИИ-агент перед началом реализации `AE3-Lite v1`.

Его задача не думать “что бы ещё улучшить”, а жёстко исполнить минимальный план без расширения scope.
Если brief конфликтует с реализационными желаниями агента, агент обязан проиграть brief, а не своим идеям.

Основной spec:
- `04_BACKEND_CORE/ae3lite.md`

---

## 1. Что именно нужно сделать

Реализуется только `AE3-Lite v1` со scope:
1. только `cycle_start`
2. только `POST /zones/{id}/start-cycle`
3. только `GET /internal/tasks/{task_id}`
4. только один runtime worker
5. только protected pipeline через `history-logger`
6. только manual rollout через `zones.automation_runtime`

Если задача не нужна для этого списка, её делать нельзя.

---

## 2. Что делать запрещено

Агенту запрещено:
1. добавлять новые task types
2. добавлять новые API кроме двух разрешённых
3. строить generic workflow engine
4. строить event bus, outbox, saga, platform abstractions
5. тащить код из `ae2lite/*` в `ae3lite/*`
6. ослаблять DB constraints ради “простоты”
7. обходить `history-logger`
8. делать direct MQTT publish
9. переходить к следующему этапу без тестов на предыдущем
10. оправдывать отклонение словами `temporary`, `quick fix`, `later`, `for now`

---

## 3. Жёсткий порядок работ

Разрешён только такой порядок:

1. Schema:
   - `ae_tasks`
   - `ae_commands`
   - `ae_zone_leases`
   - `zone_workflow_state.version`
   - `zones.automation_runtime`

2. DB invariants:
   - не более одной active task на зону
   - не более одной active lease на зону
   - уникальность `idempotency_key`
   - уникальность `(task_id, step_no)`

3. Claim/lease flow:
   - claim task
   - claim lease
   - запрет double execution

4. `CycleStartPlanner`

5. Publish gateway:
   - только `history-logger`
   - запись `external_id`

6. Reconcile:
   - только через legacy/history-logger `commands`

7. Startup recovery

8. `GET /internal/tasks/{task_id}`

9. Rollout/rollback runbook

Переход к следующему этапу запрещён, если предыдущий:
1. не завершён
2. не покрыт тестами
3. оставляет known defect в инвариантах
4. не прогнан фактическим запуском stage-level tests

---

## 4. Обязательные проверки на каждом этапе

Перед переходом дальше агент обязан доказать:
1. какой инвариант защищён
2. где именно он защищён: DB, runtime или оба слоя
3. каким тестом это подтверждено
4. какой crash/race scenario закрыт

Если доказательства нет, этап не завершён.
Если тесты этапа не запускались, этап считается непроверенным и переход запрещён.

---

## 5. Протокол DEVIATION

Если агент отклонился от плана, он обязан немедленно:
1. остановить дальнейшую реализацию
2. зафиксировать запись вида:
   `DEVIATION: <что нарушено> -> <какой пункт spec/brief нарушен>`
3. признать текущий deliverable проваленным
4. откатить собственные незамерженные изменения, вызвавшие отклонение
5. вернуться к последнему состоянию, совместимому с `ae3lite.md`
6. добавить тест, который не позволит повторить это отклонение

Повторное отклонение того же класса запрещает продолжение работы до решения человека.

---

## 6. Самонаказание агента

Под “самонаказанием” в этом проекте понимается не текстовое самоунижение, а обязательные corrective actions.

Если агент нарушил план, он обязан наказать себя так:
1. лишить себя права продолжать следующий этап
2. удалить или откатить собственное отклонение
3. потратить следующий ход только на исправление нарушения
4. добавить дополнительный тест или constraint
5. не заявлять прогресс, пока нарушение не устранено
6. заново прогнать тесты этапа после исправления

Запрещено:
1. скрывать отклонение
2. маскировать отклонение под “рефакторинг”
3. оставлять нарушение “на потом”

---

## 7. Fail-Closed Rule для агента

Если агент не уверен:
1. не расширять scope
2. не придумывать новую архитектуру
3. не писать код “на авось”
4. не ослаблять инвариант
5. предпочесть blocker вместо домысла

Если выбор стоит между:
1. “быстрее, но слабее”
2. “медленнее, но по spec”

разрешён только второй вариант.

---

## 8. Минимальный test gate

Правило этапов:
1. каждый этап обязан завершаться своими unit/integration тестами
2. без зелёных тестов этап не закрыт
3. тесты в конце всей работы не заменяют тесты по этапам
4. агент обязан сначала закрыть stage-level tests, затем идти дальше

Финальное правило:
1. после завершения всей реализации агент обязан покрыть все недостающие критические сценарии
2. после этого агент обязан прогнать полный обязательный `e2e`-набор
3. без зелёного полного `e2e` работа считается незавершённой

Агент не имеет права объявлять `AE3-Lite v1` готовым без следующих сценариев:
1. `start-cycle -> DONE -> completed`
2. `start-cycle -> TIMEOUT -> failed`
3. `restart during waiting_command -> recovered`
4. `runtime switch denied while zone busy`
5. защита от double execution на одной зоне

Если хоть один сценарий не зелёный, работа не завершена.

---

## 9. Definition of Success

Работа считается успешной только если:
1. реализован только `cycle_start`
2. protected pipeline не нарушен
3. DB invariants реально защищают runtime
4. recovery переживает restart без двойного publish
5. rollout на busy zone запрещён
6. Laravel poller читает canonical task API
7. каждый этап был проверен своими тестами до перехода дальше
8. полный обязательный `e2e`-набор покрыт и пройден
9. staging rollout и rollback воспроизводимы

Всё остальное не успех, а незавершённая работа.
