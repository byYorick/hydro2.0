# DEBUG_CHECKLIST

Практический чек-лист, чтобы Codex быстрее находил и исправлял баги в `hydro2.0`.

## 1. Поднять и проверить среду

```bash
make up
make migrate
make test
```

Если баг в runtime-потоке:

```bash
make smoke
```

## 2. Дать воспроизводимость

Перед задачей укажи:

- точные шаги воспроизведения (1..N);
- фактический результат;
- ожидаемый результат;
- частоту воспроизведения (всегда/редко/после рестарта);
- затронутую зону/ноду/идентификаторы (`zone_id`, `node_uid`, `cmd_id`).

## 3. Приложить артефакты сразу

Минимум:

- stack trace или ошибка;
- последние 100-200 строк логов нужного сервиса;
- payload/HTTP запрос/команда, которая вызвала проблему.

Полезные команды:

```bash
make logs-core TAIL=200
make logs-laravel TAIL=200
make logs-ae TAIL=200
make logs-hl TAIL=200
make logs-mqttb TAIL=200
make logs-db TAIL=200
make logs-redis TAIL=200
make logs SERVICE=<service> TAIL=200
```

## 4. Локализовать поломку тестом

- сначала добавь/запусти узкий падающий тест на баг;
- после фикса прогони этот тест и связанный набор.

Принцип: один стабильный failing test лучше длинного ручного сценария.

## 5. Использовать MCP для данных

Для быстрой диагностики состояния:

- `postgres` MCP (read-only `restricted`);
- `redis` MCP.

Конфиг: `backend/laravel/.cursor/mcp.json`.

## 6. Рекомендуемый шаблон задачи для Codex

```text
Нужно: <что сломано и что надо починить>.
Где: <сервис/файл/модуль>.
Шаги: <как воспроизвести>.
Факт: <что происходит сейчас>.
Ожидание: <что должно происходить>.
Артефакты: <логи/trace/payload/ID>.
Ограничения: <что нельзя ломать>.
```
