# ТЗ: Внедрение 1С:УНФ для сервисного центра (ремонт вендинговых запчастей + торговля)

Дата: 2026-01-05  
Версия: 1.0

## 1. Цель
Внедрить в 1С:УНФ единый контур учета и управления для:
- ремонта клиентских запчастей (3 сценария),
- подменного фонда (выдача/возврат/продажа),
- торговли запчастями (новые/БУ/из подменного фонда),
- закупки новых и БУ запчастей,
- закупки БУ аппаратов и разборки на запчасти,
- контроля денег, расходников, подменных и ремонтов по каждому ИП.

## 2. Роли и права доступа
### Роли
1) Админ/приемщик (вы): полный доступ к заказам, складам, настройкам, отчетам.  
2) Мастер (наемный, может быть несколько): доступ **только к своим заказам** (созданным/назначенным).  
3) Бухгалтер: финансы, закрывающие документы, оплаты, счета, чеки, акты.

### Правила
- Ограничение видимости заказов для мастеров: по ответственному/исполнителю.
- «Мягкие ограничения» по документам/статусам, но **жесткие по оплате**: финансовое закрытие без оплаты запрещать.
- Для клиентов «с улицы» выдача подменной только при фиксации ID.

## 3. Ключевые бизнес-правила (обязательные)
1) **1 заказ = 1 запчасть** (ремонтный заказ-наряд).
2) Клиент — отдельный контрагент: быстрый ввод *имя + телефон + источник* (обязательные).
3) Клиент закреплен за ИП; ИП выбирается **в начале заказа** и не меняется.
4) Подменные — **общий пул**, но при кросс-ИП выдаче фиксировать ответственность «кто выдал».
5) Гарантия **только на работы**: гарантийный случай — **новый заказ** со ссылкой на исходный.
6) Оплата: один тип оплаты на заказ (нал/перевод/безнал), но допускаются частичные оплаты и оплата до/после выдачи.
7) Групповая оплата партии: один платеж может закрывать несколько заказов (несколько актов/реализаций).
8) Подменная может быть не возвращена сразу после оплаты: заказ закрывается финансово, а контроль подменной продолжается.

## 4. Справочники и реквизиты
### Контрагенты
- Обязательные поля: Имя/название, Телефон, Источник (справочник).
- Опциональные: e-mail, адрес, комментарий.

### Номенклатура
- Услуги: Диагностика, Ремонт, Аренда подменной (фикс), Гарантийный ремонт (0), Прочие услуги.
- Товары:
  - Запчасти новые,
  - Запчасти БУ,
  - Подменные узлы (как товарный учет),
  - Аппараты БУ (как отдельная номенклатура),
  - Расходники (обязательно для списания на заказы).

### Серийность / идентификация
- Для подменных: обязательный внутренний ID (наклейка) и/или серийный номер.
- Для клиентских: опционально серийник; обязательно присваивать внутренний номер ремонта (например R-YYYY-NNNNNN).

## 5. Склады (структура)
Обязательные склады:
1) **Клиентские запчасти** (ответственное хранение).
2) **Ремонт ИП №1**, **Ремонт ИП №2** ... (по каждому ИП).
3) **Подменный фонд** (общий пул).
4) **Новые запчасти**.
5) **БУ запчасти**.
6) **Доноры / разборка**.

Требование: отчеты должны показывать количество и состав запчастей «в ремонте» **по каждому ИП**.

## 6. Документы и объекты учета (карта процессов → 1С)
### Ремонт
- Заказ-наряд (как «Заказ покупателя» или объект УНФ для услуг/заказов).
- Акт выполненных работ (закрытие работ).
- Оплата (Поступление ДС).

### Подменные
- Выдача подменной: фиксируется как операция движения/ответственного хранения (в УНФ — через документ движения/перемещения/спец.операцию) + запись в «Контроль подменных».
- Возврат подменной: приход обратно на склад «Подменный фонд».
- Продажа подменной: обычная реализация товара (при переводе в сценарий 3 или при невозврате).

### Торговля
- Реализация товаров (счет/чек при необходимости).
- Поступление товаров (закупка новых/БУ).
- При необходимости: резервирование.

### Закупка БУ аппарата и разбор
- Закупка аппарата: поступление на склад «Доноры / разборка».
- Разборка: разукомплектация/оприходование узлов на склады (Подменный фонд / БУ / Доноры / Списание).

## 7. Статусы заказов (единый жизненный цикл)
Рекомендуемые статусы:
1) Принято (на складе «Клиентские запчасти»)
2) В ремонте (на складе «Ремонт ИП №N»)
3) Подменная выдана (если применимо)
4) Готово
5) Выдано клиенту (операционно)
6) Ожидает оплату (финансово)
7) Закрыто (оплачено)
Дополнительно:
- Неремонтопригодно
- Гарантийный

Правило: «Выдано клиенту» и «Закрыто» разделять (операционное vs финансовое завершение).

## 8. Сценарии ремонта (подробно)
### Сценарий 1: принесли → ремонт → оплата
1) Приемка и оформление заказа.
2) Диагностика (может быть платной отдельно).
3) Ремонт + списание расходников.
4) Выдача.
5) Акт + оплата.
6) Закрытие.

### Сценарий 2: принесли → выдали подменную → ремонт → выдали отремонтированную + оплата → подменная может вернуться позже
1) Выдать подменную из «Подменного фонда» (или из другого ИП с фиксацией ответственности).
2) Зафиксировать подменную: ID, клиент, заказ, ИП, дата, срок=30 дней, платность аренды.
3) Ремонт.
4) Выдать отремонтированную.
5) Акт + оплата (заказ может быть закрыт).
6) Отдельный цикл контроля возврата подменной до фактического возврата/продажи/списания.

### Сценарий 3: принесли → выдали подменную → подменная остается у клиента → отремонтированная/клиентская остается у вас
Правила цены:
- «Только ремонт» ИЛИ
- «Продажа подменной как товара» (например, если неремонтопригодно — продажа запчасти).
Дальнейшая судьба оставшейся у вас запчасти:
- в подменный фонд (если восстановлена),
- в БУ склад,
- в доноры,
- списание.

### Неремонтопригодно / клиент оставил
Два варианта:
- Оставил бесплатно (смена собственности).
- Выкуп у клиента (закупка/поступление).
Далее — E2 (разбор/оприходование).

## 9. Контроль подменных (обязательная подсистема)
Требуется реестр/механизм контроля «Подменные на руках» (не зависящий от статуса оплаты заказа):
Поля:
- ID подменной
- Клиент
- Заказ
- ИП (исполнитель) + ИП (выдал) при кросс-ИП
- Дата выдачи
- Плановая дата возврата (выдача + 30 дней)
- Признак аренды (платно/бесплатно) + сумма (фикс)
- Фактическая дата возврата
- Статус: на руках / просрочено / возвращено / продано / списано

Операции:
- Выдача (создание записи)
- Возврат (закрытие записи)
- Просрочка (отбор в отчете)
- Конвертация в продажу (создание реализации из подменного фонда)
- Списание/утеря (документ списания + причина)

## 10. Расходники и себестоимость
- Списание расходников обязательно на заказ (для аналитики и себестоимости).
- Отчеты должны показывать расход по периодам и по заказам/ИП.
- Для сценария 3 учитывать себестоимость при перемещениях/продаже.

## 11. Печатные формы (шаблоны)
Нужны шаблоны (печать опциональна):
- Акт приемки
- Квитанция
- Счет
- Чек
- Акт выполненных работ

## 12. Отчеты (обязательные)
1) Ремонты по статусам (в ремонте/готово/ожидает оплату/закрыто).
2) **Запчасти в ремонте по каждому ИП** (по складам «Ремонт ИП №N»).
3) Подменные на руках + просроченные > 30 дней.
4) Заказы закрыты, но подменная не возвращена.
5) Выручка по ИП, неоплаченные/частично оплаченные.
6) Расходники: расход по позициям/заказам/ИП.
7) Гарантийные: список гарантийных заказов с ссылкой на исходный.

## 13. Не-функциональные требования
- Простота ввода (быстрый контрагент, быстрый заказ).
- Минимум обязательных полей, кроме критичных (ID подменной для «улицы»).
- Масштабирование на несколько мастеров.
- Единый стандарт внутренних номеров (ремонт/подменные).

## 14. Критерии приемки
1) Можно провести все 3 сценария ремонта end-to-end.
2) Можно закрыть заказ по оплате при невозвращенной подменной (сценарий 2) и видеть это в отчетах.
3) Можно продать из подменного фонда как товар.
4) Можно купить БУ аппарат, разобрать и получить запчасти на нужные склады.
5) В любой момент видно количество запчастей в ремонте по каждому ИП.
6) Работает контроль расходников.
7) Права доступа: мастер видит только свои заказы.
