+# Обзор WebSocket-архитектуры Hydro (RU)
+
+## Бэкенд: вещание событий
+- Laravel Broadcasting использует Reverb по умолчанию; параметры подключения (host/scheme/ports/path, ключи приложения, разрешённые origins) берутся из переменных окружения.
+- Все серверные события публикуются в приватные каналы: обновления и ошибки команд (`commands.{zoneId}` или `commands.global`), новые события зон (`events.global`), оповещения (`hydro.alerts`), а также изменения зон (`hydro.zones.{id}`).
+- Доступ к приватным каналам выдаётся через маршрут `/broadcasting/auth` под middleware `web` + `auth`, который вызывает `Broadcast::auth(request())`.
+- Отправка выполняется через события и наблюдатели моделей: `CommandObserver` транслирует обновления/ошибки команд, `ZoneEventObserver` публикует новые события зон; каждое событие описывает имя канала и полезную нагрузку.
+- В health-эндпоинте статус WebSocket помечается как «unknown», чтобы фронтенд мог самостоятельно проверять соединение.
+
+## Фронтенд: инициализация Echo
+- Модуль `resources/js/utils/echoClient.ts` собирает конфиг Echo из Vite-env (протокол/хост/порт/путь/auth-заголовки), проверяет `VITE_ENABLE_WS`, создаёт единственный экземпляр Echo в `window`, навешивает слушатели состояний Pusher и планирует экспоненциальные реконнекты с логированием ошибок.
+- Вспомогательный `bootstrap.js` гарантирует единоразовую инициализацию через `initEcho`, делает дополнительные проверки после загрузки страницы, отключает соединение при `beforeunload/pagehide`, при возврате страницы переинициализирует клиент.
+
+## Подписки и мониторинг на фронте
+- Компосабл `useWebSocket` повторно использует приватные каналы (команды по зоне, глобальные события), отслеживает «мертвые» каналы, корректно пересоздаёт подписки, предотвращает дубли и отписывает каналы при размонтировании компонентов.
+- Компосабл `useSystemStatus` читает состояние Pusher прямо из `window.Echo`, показывает статусы `connected/connecting/disconnected`, не конфликтует с инициализацией в `bootstrap.js` и отдаёт сведения о попытках реконнекта и последней ошибке.
+
+## Предложения по улучшению (фокус на System Status)
+- **Сделать статус WebSocket более точным в API health-check**: возвращать фактическое состояние подключения Reverb/Redis (например, успешный ping к Reverb порту и проверку очереди `broadcasts`) вместо «unknown», чтобы фронтенд мог показывать «недоступно», когда сервер реально недостижим.
+- **Добавить явный флаг деградации в `useSystemStatus`**: вычислять `degraded`, если попыток реконнекта больше заданного порога или последнее событие `error` свежее N секунд, и показывать в UI причину (HTTP-код auth, timeout, network). Это уменьшит «ложно зелёные» статусы.
+- **Интегрировать тестовую подписку/пинг в bootstrap**: при инициализации Echo подписываться на небольшой публичный канал `ws.health` или отправлять ping событие, чтобы сразу выявлять конфликты CORS/SSL ещё до пользовательских действий.
+- **Улучшить телеметрию соединений**: логировать метаданные реконнектов (протокол, задержку backoff, последние ошибки) в `useSystemStatus` и прокидывать их в систему метрик/bug reports для быстрой диагностики нестабильных сетей.
